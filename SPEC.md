---

## 【目标】

做一个 **openclaw 活动监控器（Activity Monitor）**：实时读取 openclaw 的会话日志文件，将其中的结构化 JSON 数据解析并转换为人类可读的文本流，让用户能随时清晰了解 openclaw 正在做什么、遇到了什么情况、做出了什么决策。

---

## 【技术栈】

- 语言：Node.js（v18+）
- 运行方式：CLI 命令行工具（无需框架）
- 核心依赖：
  - `fs`（原生）：文件读取与 tail 监听
  - `readline`（原生）：逐行解析
  - `chalk`（可选）：终端彩色输出
  - `date-fns` 或原生 `Date`：时间格式化
- 测试：Jest
- 无需数据库、无需网络请求

---

## 【需求细节】

1. **实时 Tail 监听**：程序启动后，持续监听指定日志文件（路径通过 CLI 参数传入），类似 `tail -f`，有新行写入时立刻处理。

2. **JSON 行解析**：每行日志为一条独立 JSON 对象（JSONL 格式）。程序逐行读取，解析 JSON，解析失败的行跳过并在 stderr 打印警告。

3. **事件类型识别与翻译**：根据 JSON 中的 `type`（或 `event`）字段，将不同类型的事件翻译为人类友好的中文/英文描述，至少支持以下类型：

   | 事件类型（type）       | 展示文案示例                              |
   |----------------------|----------------------------------------|
   | `tool_use`           | 🔧 正在使用工具：`bash` → `ls -la`        |
   | `tool_result`        | ✅ 工具返回：（截断展示前200字符）           |
   | `thinking`           | 💭 正在思考：（展示思考摘要或前100字符）      |
   | `message`            | 💬 消息：（展示 content 内容）             |
   | `error`              | ❌ 遇到错误：（展示 error.message）        |
   | `decision`           | 🧭 决策：（展示决策内容）                  |
   | `file_read`          | 📂 读取文件：`<路径>`                    |
   | `file_write`         | 💾 写入文件：`<路径>`（字节数）             |
   | `unknown`（兜底）     | ℹ️  未知事件 [type]：（JSON 原文摘要）     |

4. **输出格式**：每条事件以统一格式输出到 stdout：
   ```
   [HH:MM:SS] <emoji> <人类可读描述>
   ```
   长内容自动截断（默认200字符），并附加 `...（已截断）` 提示。

5. **CLI 参数支持**：
   - 必填：`--log <path>` 或第一个位置参数，指定日志文件路径
   - 可选：`--lang <zh|en>` 输出语言，默认 `zh`
   - 可选：`--max-len <number>` 内容截断长度，默认 200
   - 可选：`--filter <type1,type2>` 只显示指定事件类型
   - 可选：`--no-color` 禁用彩色输出

6. **历史回放**：启动时先读取文件中已有的所有行（回放历史），再进入实时监听模式，并在历史/实时之间打印分隔线：
   ```
   ──────────── 历史回放结束，进入实时监听 ────────────
   ```

7. **优雅退出**：Ctrl+C 时打印 `\n监控已停止。` 并正常退出（exit code 0）。

---

## 【约束条件】

- 不得依赖任何需要系统级安装的外部工具（如 `jq`、`python` 等）
- 必须兼容 Node.js 18+，不使用实验性 API
- 日志文件不存在时，打印友好错误提示后退出（exit code 1），不崩溃
- 不得将日志内容上传到任何网络端点，全程本地处理
- 单文件或极少文件实现（入口 `monitor.js` + 可选 `parser.js`），不引入复杂工程结构
- 内存占用友好：不将整个日志文件载入内存，逐行流式处理

## 【验收标准】（必填，Agent 依此判断是否完成）

- **AC1**：`npm test` 全部通过，覆盖率 > 80%（覆盖 JSON 解析、事件翻译、截断逻辑）
- **AC2**：`node monitor.js --log ./test/sample.jsonl` 能正常启动，输出历史行后进入监听，无报错
- **AC3**：向 `sample.jsonl` 追加一行 `tool_use` 类型 JSON 后，终端在 1 秒内输出对应的可读文本
- **AC4**：追加一行非法 JSON（如 `{broken`）时，stderr 打印警告，stdout 不中断，程序继续运行
- **AC5**：`--filter tool_use,error` 参数生效，其他类型事件不输出到 stdout
- **AC6**：`--log` 指向不存在的文件时，输出 `错误：文件不存在 <path>`，exit code 为 1
- **AC7**：`--no-color` 时输出纯文本，无 ANSI 转义码

## 【补充说明】

- **日志格式假设**：每行一个 JSON 对象（JSONL），字段结构参考：
  ```json
  { "timestamp": "2025-01-01T12:00:00Z", "type": "tool_use", "tool": "bash", "input": "ls -la" }
  { "timestamp": "2025-01-01T12:00:01Z", "type": "tool_result", "content": "total 32\n..." }
  { "timestamp": "2025-01-01T12:00:02Z", "type": "thinking", "content": "需要先确认目录结构..." }
  { "timestamp": "2025-01-01T12:00:03Z", "type": "error", "error": { "message": "Permission denied" } }
  ```
  如实际 openclaw 日志字段不同，`parser.js` 应设计为易于修改映射关系。

- **扩展性**：`parser.js` 中事件类型映射应写成配置对象，便于后续添加新类型，无需修改核心逻辑。

- **参考行为**：整体交互体验类似 `docker logs -f` 或 `kubectl logs -f`，简洁、实时、可读。
